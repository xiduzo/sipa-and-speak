## Current State

No test suite exists yet. This document establishes the standards to build toward. Until tests are added, the sole automated quality gate is `bun check-types`. All new code should be written with pure functions and explicit side-effect boundaries to make future coverage low-friction to add.

## Test Strategy

Target stack once tests are introduced:

| Level | Type | Tool | Purpose |
| --- | --- | --- | --- |
| Unit | Pure functions, tRPC input validators | Vitest | Validate business logic in isolation — matching rules, profile validators, Zod schemas |
| Integration | tRPC router + Drizzle + real DB | Vitest + real PostgreSQL | Validate that API procedures produce correct DB state |
| E2E | Critical user flows | Playwright (web) / Detox (native, future) | Validate full flows: onboarding → matching → meetup scheduling |

**Rationale for Vitest:** native ESM support, Bun-compatible, zero-config for TypeScript monorepos.

## Coverage Requirements

No hard threshold is enforced yet. Focus on critical paths first:

- **Matching logic** (`packages/api/src/routers/matching.ts`) — must have unit tests before any changes
- **Auth flows** (`packages/auth/`) — must have integration tests before any changes; the Expo + Better-Auth integration is non-standard and high-risk
- **Meetup scheduling** (`packages/api/src/routers/meetup.ts`) — happy path and conflict cases
- **Bug fixes** must include a regression test that would have caught the bug

Once the suite reaches meaningful coverage, enforce **80% lines + branches** on new and modified files via Vitest thresholds.

## Test Patterns

- **Framework:** Vitest (to be configured at root and per-package as tests are added)
- **Test file location:** Colocated with source — `src/foo.ts` → `src/foo.test.ts`
- **Fixture strategy:** Factory functions in a `tests/factories/` folder per package
- **Mock boundary:** Only mock at system boundaries (external APIs, email provider, Polar). Do not mock tRPC routers, Drizzle, or internal application code.
- **Database:** Integration tests must run against a real PostgreSQL instance (`bun db:start`). No in-memory fakes.
- **Auth in tests:** Use test-specific Better-Auth sessions; never share session state between tests.

## Gherkin → Test Mapping

When tasks include Gherkin scenarios (Given/When/Then), each scenario maps to at least one automated test:

- Test names reference the scenario name so failures are traceable
- `Given` steps → test setup (`beforeEach` / fixtures)
- `When` steps → the action under test
- `Then` steps → assertions
- Scenarios describe observable behaviour, not implementation details

## Definition of Done

A task is ready to merge when:

- [ ] `bun check-types` passes with zero errors across all packages
- [ ] All previously passing tests still pass (no regressions introduced)
- [ ] Any new logic in a critical path (matching, auth, meetup scheduling) has at least one test
- [ ] Bug fixes include a regression test
